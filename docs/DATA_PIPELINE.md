# 전국 문제 자동 생성 파이프라인

## 왜 완전 실시간 랜덤보다 “검증된 랜덤 문제 팩”인가

전국 버스 데이터는 지역별 ID와 갱신 시점이 다르고, 철도·도시철도·시내버스를 한 그래프로 합치는 순간 환승역 연결과 도보 링크가 필요합니다. 임의 좌표 두 개를 즉석에서 뽑으면 정류장이 없거나, 섬·산악·군사시설에 떨어지거나, API가 일부 교통수단을 빠뜨려 잘못된 정답을 만들 수 있습니다.

권장 방식은 다음과 같습니다.

```text
정류장/역 수집 → 본토 후보점 생성 → 거리 난이도 필터 → 경로 후보 계산
→ 1km 도보 규칙 검증 → 최소환승 재정렬 → 사람 검수 → 문제 스냅샷 배포
```

## 1. 후보점 생성

- 행정구역 본토 폴리곤 안의 좌표만 사용합니다. 제주·울릉도와 교량 없이 연결되지 않은 섬은 제외합니다.
- 임의 좌표부터 만들지 말고, 운영 중인 정류장/역을 먼저 뽑은 뒤 반경 50~700m 안의 도로 접근 가능한 지점을 출발·도착점으로 만듭니다.
- 시작·도착 각각 가장 가까운 승차 지점까지 실제 보행 경로가 1km 이하여야 합니다. 직선거리만으로 판정하지 않습니다.
- 학교·군부대·고속도로 본선처럼 보행 출입이 불가능한 지점은 제외합니다.

## 2. 거리 난이도

현재 UI의 권장 구간은 다음과 같습니다.

- 10km: 직선거리 3~20km
- 50km: 20~70km
- 100km+: 70km 이상
- 전범위: 전체 문제 팩에서 무작위

난이도는 거리만으로 끝나지 않습니다. 같은 거리에서도 직통 KTX는 0회, 도시 외곽 버스는 3회일 수 있으므로 향후에는 `거리 × 교통망 밀도 × 정답 환승 수`로 별도 별점도 계산하는 편이 좋습니다.

## 3. 경로 정규화

각 후보 경로를 아래 공통 형태로 바꿉니다.

```ts
type NormalizedLeg = {
  mode: "walk" | "bus" | "subway" | "train";
  serviceId?: string;
  fromStopId: string;
  toStopId: string;
  distanceMeters: number;
  geometry: [number, number][];
};
```

규칙:

1. 모든 `walk` leg의 실제 보행거리가 각각 1,000m 이하여야 합니다.
2. 택시·자가용·항공·자전거가 포함된 경로는 탈락시킵니다.
3. 동일 차량이 노선번호만 바뀌지 않고 계속 운행하는 경우는 새 승차로 세지 않습니다.
4. `최소 환승 횟수 = 대중교통 승차 leg 수 - 1`입니다.
5. 환승 수가 같은 경로가 여럿이면 총 보행거리, 총 시간 순으로 대표 해설 경로를 고릅니다.
6. 도시 간 ODsay 결과에는 앞뒤 도시 내 접근 경로를 결합한 뒤 다시 계산합니다.

## 4. 검증

- 출발·도착 이름과 도로명주소를 역지오코딩 결과로 저장합니다.
- TAGO/지역 API에서 각 노선이 현재 운행 중인지 확인합니다.
- ODsay 결과의 `busTransitCount`, `subwayTransitCount`, 세부 `subPath`를 단순 합산하지 말고 정규화된 승차 leg로 다시 계산합니다.
- 정답 후보가 1개뿐인 문제보다 서로 다른 공급자/시간대에서 같은 최소환승 값이 확인된 문제를 우선합니다.
- `verified_at`, `source_version`, `reviewer`, `expires_at`을 저장합니다.
- 개편으로 만료된 문제는 즉시 비활성화하고 클라이언트 문제 팩에서 제외합니다.

## 5. 치팅 방지

정적 GitHub Pages 번들에 `minimumTransfers`가 들어 있으면 개발자 도구로 볼 수 있습니다. 캐주얼 모드는 현재 구조로 충분하지만 경쟁 모드는 다음처럼 분리합니다.

- 클라이언트가 받는 문제: ID, 출발·도착 좌표/주소, 제한시간, 공개 교통정보만 포함
- 서버가 보관하는 정보: 정답, 전체 경로, 만료일, 점수 계산식
- 제출 시 클라이언트: `questionId`, 추측값, 남은 시간, 라운드 토큰 전송
- Edge Function: 정답 조회 → 점수 계산 → 결과와 해설 경로 반환 → 리더보드 저장

클라이언트가 보낸 점수를 그대로 DB에 INSERT하면 조작이 가능하므로 금지합니다. `supabase/functions/submit-score` 예제는 서버가 문제 정답을 다시 조회해 점수를 계산합니다.

## 6. 추천 DB 필드

- `game_questions`: 좌표, 주소, 거리 구간, 최소 환승, 공개 여부, 검증 시각
- `question_legs`: 순서, 수단, 노선, 출도착 정류장, 시간, 거리, GeoJSON
- `transit_stops`: 공급자별 ID, 좌표, 유형, 운행 상태
- `transit_services`: 노선 ID, 명칭, 교통수단, 운영기관
- `leaderboard_scores`: 서버 계산 점수, 닉네임, 난이도, 정확 답 개수

전국 정류장 근접 검색이 커지면 Supabase의 PostGIS 확장을 켜고 `geography(Point, 4326)` + GiST 인덱스를 사용하는 것이 좋습니다.
